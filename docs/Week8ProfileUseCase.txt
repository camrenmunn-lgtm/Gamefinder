Use Case Narrative — Manage User Profile
Use Case Name: Manage User Profile
Actor: Authenticated User (Logged-in Student)
Goal: View and update personal profile information, change password, or delete account
Preconditions: The user is actively logged into the application. The user navigates to /users/profile.

Main Success Scenario — Profile Update

The user clicks "Edit Profile" in the top navigation bar.
The system retrieves the user's current data from the database using their email (from the Spring Security session).
The system formats the phone number for display (e.g., 3199999999 → (319) 999-9999) and clears the password field for security.
The system renders the profile form pre-populated with the user's current data.
The user modifies one or more fields (First Name, Last Name, Nickname, Email, Phone, Preferred Language, Public Email/Phone toggles) and submits the form.
The system validates the input using bean validation rules.
If the email was changed, the system checks whether the new email is already registered to another account.
If a new password was entered, the system validates it against the strength requirements.
The system strips non-numeric characters from the phone number before saving.
The system saves the updated record to the database.
If the email changed, the system refreshes the Spring Security context so the session reflects the new email.
The system redirects the user back to /users/profile with a success flash message: "Your profile has been updated successfully."


Alternative Flow A — Validation Fails

At step 6, if any field fails validation (e.g., invalid email format, weak password), the system re-renders the profile form with field-level error messages. No changes are saved to the database.


Alternative Flow B — Duplicate Email

At step 7, if the new email is already registered to another account, the system adds an error to the email field: "This email is already taken." The form is re-rendered and no changes are saved.


Alternative Flow C — Password Change

At step 5, if the user enters a value in the password field, the system applies the strength validation regex (min 8 chars, uppercase, lowercase, number).
If valid, the system BCrypt-hashes the new password before saving.
If the password field is left blank, the existing password is preserved unchanged.


Alternative Flow D — Account Deletion

The user clicks the "Delete Account" button in the Danger Zone section.
The browser displays a JavaScript confirmation dialog asking the user to confirm.
If confirmed, the system performs a soft delete by setting deleted_at to the current timestamp and scrambling the email address to free it for re-registration.
The system logs the user out by invalidating their HTTP session using SecurityContextLogoutHandler.
The system redirects the user to the homepage with a farewell flash message: "Your account has been successfully deleted. We're sorry to see you go!"
If the user attempts to log in again with the same credentials, the UserDetailsService detects the non-null deleted_at and rejects the login attempt.
